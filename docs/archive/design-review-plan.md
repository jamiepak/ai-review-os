### 模块①：需求收集与整理

#### 功能描述
- 录入 TAPD 链接（支持任意格式粘贴）
- 自动解析 TAPD 需求信息（名称、概述、创建人、设计师等）
- 写入企微智能表格，形成评审 Backlog

#### 技术方案

| 组件 | 技术实现 | 状态 |
|------|---------|------|
| 前端页面 | 移动端 HTML 页面，支持粘贴 TAPD 链接 | ✅ 已完成 |
| TAPD 解析 | 通过 TAPD MCP API（stories_get）提取需求字段 | ✅ 已验证 |
| 写入表格 | 企微智能表格 Webhook（add_records） | ✅ 已验证 |
| 代理服务 | 腾讯云函数 + 函数 URL（解决 CORS） | ✅ 已部署 |

#### 数据流

```
用户粘贴链接 → 前端提取需求ID → 云函数代理 → TAPD API 查询
                                                    ↓
企微智能表格 ← 云函数代理 ← 前端组装数据 ← 返回需求详情
```

### 模块②：评审过程实时打分

#### 功能描述
- 评委在评审过程中实时录入评审结果
- 支持「通过 / 不通过 / 待修改」结论
- 支持添加评审意见/问题记录
- 结果实时写入智能表格

#### 技术方案

| 方案 | 实现方式 | 优劣 |
|------|---------|------|
| **A. 移动端评分页面（推荐）** | 新建评审打分页面，展示待评审列表，评委点选结论 | 操作直觉，适合会中使用 |
| B. 对话式打分 | 通过 WorkBuddy 对话指令输入评审结论 | 灵活但需逐条操作 |

#### 方案 A 详细设计

**页面功能：**
1. 展示当天/指定日期的待评审需求列表（需从表格读取）
2. 每条需求支持：
   - 「通过」「不通过」「待修改」三按钮
   - 评审意见文本框
   - 确认提交
3. 提交后通过 Webhook 更新对应记录的「评审结论」字段

**核心难点：**
- ⚠️ **读取现有记录**需要 access_token 模式（Webhook 无法查询）
- ⚠️ **更新记录**需要知道 record_id

**两种解决方案：**

| 方案 | 实现 | 依赖 |
|------|------|------|
| **A1: 全 API 模式** | 用 access_token 调用查询记录 + 更新记录 API | 需 corpid + corpsecret |
| **A2: Webhook + 前端缓存** | 创建时记住 record_id，评分时用 Webhook 的 update_records | 仅需 Webhook key，但需前端维护 record_id 映射 |

**推荐 A2**（Webhook + 前端缓存）：
- 模块①创建记录时，把返回的 record_id 存到 localStorage / 云函数侧
- 评分页面直接通过 record_id 用 Webhook 更新

#### 技术可行性：✅ 可行

| 能力 | 状态 |
|------|------|
| Webhook 更新记录 | ✅ 已验证 |
| 前端评分界面 | ✅ 可开发 |
| record_id 管理 | ✅ 方案A2可实现，无需额外权限 |

### 模块③：评审归档与复盘

#### 功能描述
1. 每场评审结束后，自动从腾讯会议提取录制视频
2. 将评审片段与需求绑定
3. 产出评审概述和结论
4. 支持将历史评审数据导出并进行复盘分析
5. 总结评审成功/失败的模式

#### 技术方案

##### 3a. 腾讯会议录制获取

| 能力 | API | 状态 |
|------|-----|------|
| 查询录制列表 | `GET /v1/records` | ⚠️ 需企业版+开发者权限 |
| 查询录制详情 | `GET /v1/records/{id}` | ⚠️ 同上 |
| 录制转写（语音→文字） | `GET /v1/records/transcripts/{id}` | ⚠️ 同上 |
| AI 会议摘要 | `GET /v1/meetings/{id}/ai-summary` | ⚠️ 需商业版/企业版 |
| AI 会议纪要 | `GET /v1/meetings/{id}/ai-minutes` | ⚠️ 需商业版/企业版 |

**关键依赖：**
- 🔑 需要腾讯会议开放平台的 **AppId + SecretKey**
- 🔑 AI 摘要/纪要能力需要**商业版或企业版**授权
- 签名方式：HMAC-SHA256

##### 3b. 视频片段与需求绑定

| 方案 | 实现方式 | 可行性 |
|------|---------|--------|
| **时间戳匹配（推荐）** | 通过转写文本，用关键词（需求名称）搜索定位时间戳区间 | ✅ 可行，依赖转写 API |
| 语义匹配 | 用 AI 对转写内容做语义分段，匹配需求描述 | ✅ 可行，精度更高但需额外 AI 调用 |

**流程：**
```
腾讯会议录制 → 获取转写文本（带时间戳）
                     ↓
              按需求关键词分段匹配
                     ↓
              每个需求 → 对应视频时间区间 + 转写片段
                     ↓
              AI 生成该需求的评审概述和结论
```

##### 3c. 历史复盘分析

| 能力 | 实现 |
|------|------|
| 导出所有评审数据 | 从智能表格 API 批量查询（需 access_token） |
| 分析成功/失败模式 | AI 对历史评审数据做模式识别和总结 |
| 生成报告 | WorkBuddy 自动生成 Markdown/HTML 报告 |

#### 技术可行性

| 子功能 | 可行性 | 卡点 |
|--------|--------|------|
| 获取录制/转写 | ⚠️ **条件可行** | 需腾讯会议开放平台凭证 |
| AI 摘要/纪要 | ⚠️ **条件可行** | 需商业版/企业版 |
| 视频片段绑定 | ✅ 可行 | 依赖转写 API |
| 评审概述生成 | ✅ 可行 | AI 处理转写文本 |
| 历史复盘 | ⚠️ **条件可行** | 需 corpid/corpsecret 读表格 |

### 模块④：AI 预评审 Skill

#### 功能描述
1. 基于历史评审数据，训练/总结评审标准
2. 新需求上传后，自动做预评审
3. 输出预评审意见：可能通过/不通过的点、建议改进方向

#### 技术方案

##### 4a. 评审知识库构建

```
历史评审数据（表格+转写+结论）
          ↓
    AI 分析总结
          ↓
    评审 Checklist / 评审标准文档
          ↓
    存储为 WorkBuddy Skill（knowledge base）
```

**知识库内容：**
- 常见评审不通过原因分类（交互问题/视觉问题/逻辑问题/...）
- 评审通过的共性特征
- 各 FT/项目组的评审偏好
- 评审官的关注重点

##### 4b. 预评审流程

```
新需求（TAPD 链接）→ 提取需求信息 + 设计稿
                          ↓
                   AI 对照评审 Checklist 评估
                          ↓
                   输出预评审报告：
                   - 风险点标注
                   - 可能的评审问题
                   - 改进建议
                   - 预测通过率
```

##### 4c. Skill 结构

```
review-copilot/
├── SKILL.md              # Skill 入口（评审标准 + 预评审指引）
├── knowledge/
│   ├── checklist.md      # 评审 Checklist
│   ├── patterns.md       # 历史评审模式总结
│   └── ft_profiles.md    # 各FT评审特征
└── scripts/
    └── pre_review.sh     # 预评审执行脚本
```

#### 技术可行性：✅ 可行

| 能力 | 状态 |
|------|------|
| AI 文本分析 | ✅ WorkBuddy 内置 |
| 知识库构建 | ✅ Skill 框架支持 |
| TAPD 数据获取 | ✅ 已实现 |
| 预评审输出 | ✅ AI 生成 |
| 持续学习更新 | ✅ 每次评审后更新 knowledge 文件 |
