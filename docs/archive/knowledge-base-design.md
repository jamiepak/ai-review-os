# 评审知识库设计方案

> 日期: 2026-04-14 | 项目: AI Review OS · 沉淀层

---

## 一、知识库的本质

评审知识库不是一个数据库，而是一个**持续进化的评审大脑**：

```
原始数据（评审记录、转写、结论）
         ↓ 结构化
结构化知识（模式、规律、标准）
         ↓ 抽象
可执行规则（Checklist、预评审 Skill）
         ↓ 反馈
持续更新（每场评审后自动迭代）
```

---

## 二、知识库架构

```
review-knowledge/
│
├── 📊 数据层（raw/）          ← 每场评审的原始结构化记录
│   └── sessions/
│       ├── 2026-04-14_session_001.md
│       ├── 2026-04-21_session_002.md
│       └── ...
│
├── 📖 知识层（knowledge/）    ← AI 从数据中提炼的规律和标准
│   ├── checklist.md           ← 评审 Checklist（核心）
│   ├── failure_patterns.md    ← 失败模式库
│   ├── success_patterns.md    ← 成功模式库
│   ├── reviewer_insights.md   ← 评审官偏好和高频反馈
│   └── domain_profiles.md     ← 各 FT/项目组画像
│
├── 🤖 规则层（rules/）        ← 可直接执行的评审规则
│   ├── pre_review_rules.md    ← 预评审规则集
│   ├── risk_indicators.md     ← 风险信号指标
│   └── quality_gates.md       ← 质量门禁标准
│
└── 📈 索引层（index/）        ← 检索和统计
    ├── stats.md               ← 评审统计概览
    ├── tag_index.md           ← 标签索引（按类型/FT/结论）
    └── search_index.md        ← 语义搜索索引
```

## 四、知识层设计（AI 提炼）

### 4.1 评审 Checklist（核心文件）

```markdown
# 设计评审 Checklist

> 最后更新: 2026-04-14 | 基于 N 场评审数据

## 通用项（所有需求必查）

### 🔴 P0 - 必须满足
- [ ] 需求目标明确，有可量化的成功指标
- [ ] 核心交互流程完整，无逻辑断裂
- [ ] 异常态/边界场景有明确设计（空状态、错误态、loading态）
- [ ] 数据支撑充分（用户调研/数据分析/竞品参考）

### 🟡 P1 - 建议满足
- [ ] 有竞品对比分析
- [ ] 方案有 A/B 对比或替代方案说明
- [ ] 对上下游影响有评估
- [ ] 无障碍/国际化考虑

### 🟢 P2 - 加分项
- [ ] 有原型/动效 demo
- [ ] 有用户测试反馈
- [ ] 方案可扩展性论述

## 按评审类型的专项检查

### 交互评审专项
- [ ] 操作路径 ≤ 3 步（核心场景）
- [ ] 反馈及时性（操作后 200ms 内有响应）
- [ ] 撤销/回退机制

### 视觉评审专项
- [ ] 符合设计规范/Design Token
- [ ] 多分辨率适配
- [ ] 动效合理（不过度）
```

**更新机制**：每次评审结束后，AI 分析本场评审意见，提取新的检查项，更新 Checklist。

### 4.2 失败模式库

```markdown
# 评审失败模式库

> 基于历史 N 场评审中被否决/待修改的案例

## 模式 1: 异常态缺失（出现频率: 35%）
- **表现**: 只设计了正常流程，未考虑空状态/错误态/弱网/权限不足等场景
- **典型评审反馈**: "断网了怎么办？" "数据为空时展示什么？"
- **典型受影响FT**: 基础体验、企微专项
- **建议**: 设计稿中必须包含至少 3 种异常态页面

## 模式 2: 缺数据支撑（出现频率: 28%）
- **表现**: 方案基于主观判断，无用户数据/调研/竞品分析支撑
- **典型评审反馈**: "这个结论的数据依据是什么？" "用户真的需要这个吗？"
- **建议**: 必须包含至少一项数据支撑（埋点数据/用户访谈/竞品截图）

## 模式 3: 流程过长（出现频率: 18%）
...

## 模式 4: 方案唯一（出现频率: 15%）
...
```

## 五、规则层设计（可执行的预评审规则）

### 5.1 预评审规则集

```markdown
# 预评审规则集

> 这些规则用于 AI 自动预评审，在需求提交后、正式评审前执行

## 规则格式
每条规则包含: 检查条件 → 风险等级 → 建议

## 规则列表

### R001: 需求描述长度检查
- 条件: description 字数 < 100
- 风险: 🟡 中
- 建议: "需求描述过短，建议补充背景、目标和关键场景"

### R002: 缺少设计稿链接
- 条件: 设计稿地址为空
- 风险: 🔴 高
- 建议: "缺少设计稿，评审可能无法有效进行"

### R003: 无数据支撑关键词
- 条件: description 中不包含 ["数据显示", "调研", "用户反馈", "竞品", "DAU", "转化率", "%"] 中任何一个
- 风险: 🟡 中
- 建议: "未发现数据支撑内容，建议补充相关数据"

### R004: 历史相似需求匹配
- 条件: 与知识库中某条失败案例语义相似度 > 0.8
- 风险: 🔴 高
- 建议: "发现与历史被否案例相似: [案例名称]。被否原因: [原因]。建议提前规避。"

### R005: FT 特有风险检查
- 条件: 根据所属 FT 的画像，检查该 FT 的高频打回原因是否已被覆盖
- 风险: 🟡 中
- 建议: 动态生成
```

### 5.2 质量门禁

```markdown
# 评审质量门禁

## 提交门禁（进入评审池的最低要求）
- ✅ 有 TAPD 链接
- ✅ 需求描述 > 100 字
- ✅ 有设计稿或原型链接
- ✅ 指定了所属 FT

## 评审门禁（建议评审通过的标准）
- ✅ P0 Checklist 全部满足
- ✅ 预评审无 🔴 高风险项（或已有解决说明）
- ✅ 评审意见已全部回复
```

## 七、知识库更新机制

### 自动更新流程

```
每场评审结束
      ↓
1. 收集本场数据（结论、意见、转写）
      ↓
2. 生成本场评审记录（→ raw/sessions/）
      ↓
3. AI 分析本场数据:
   - 提取新的失败/成功模式 → 更新 failure/success_patterns.md
   - 检查是否有新的 Checklist 项 → 更新 checklist.md
   - 更新 FT 画像数据 → 更新 domain_profiles.md
   - 更新统计数据 → 更新 index/stats.md
      ↓
4. 如果发现新规则 → 更新 rules/pre_review_rules.md
      ↓
5. 通知: "知识库已更新，新增 X 条模式，更新 Y 条规则"
```

### 更新频率

| 内容 | 更新频率 | 触发方式 |
|------|---------|---------|
| 评审原始记录 | 每场评审后 | 自动 |
| 失败/成功模式 | 每场评审后 | AI 自动分析 |
| Checklist | 每月/每 10 场 | AI 自动 + 人工审核 |
| FT 画像 | 每月 | AI 自动 |
| 预评审规则 | 每月 | AI 建议 + 人工确认 |
| 统计数据 | 每场评审后 | 自动 |

## 八、实现路径

### Phase 1（现在可做 · 无额外依赖）

手动冷启动知识库：

1. **创建 Skill 骨架** → `review-knowledge/` 目录结构
2. **初版 Checklist** → 基于你和团队的经验手写第一版
3. **初版规则** → 基于常识写 5-10 条预评审规则
4. **接入预评审** → 提交需求时自动跑规则检查

### Phase 2（需评审数据积累 · 5-10 场后）

数据驱动迭代：

1. **自动生成评审记录** → 从智能表格导出 + 评审意见
2. **AI 分析模式** → 提炼失败/成功模式
3. **更新 Checklist** → AI 建议 + 人工审核

### Phase 3（需腾讯会议 API · 有转写数据后）

深度知识挖掘：

1. **会议转写入库** → 关键讨论片段绑定需求
2. **语义搜索** → "类似 XX 的需求以前怎么评审的？"
3. **AI Reviewer** → 评审过程中实时匹配知识库给建议

## 九、知识库的使用场景

| 场景 | 用法 | 阶段 |
|------|------|------|
| 提交需求时 | AI 自动跑预评审规则，输出风险报告 | Phase 1 |
| 评审前 | AI 生成一页纸 summary + 建议关注点 | Phase 1 |
| 评审中 | AI 实时匹配知识库提示风险 | Phase 3 |
| 评审后 | AI 自动归档 + 更新知识库 | Phase 2 |
| 写需求时 | AI 提示"这个需求类似 XX，当时被否原因是 XX" | Phase 2 |
| 季度复盘 | AI 生成评审质量报告（趋势、改进点） | Phase 2 |
| 新人入职 | 通过知识库快速了解评审标准和历史 | Phase 2 |
